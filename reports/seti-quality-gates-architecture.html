SETI / ARCHITECTURE NOTE / QUALITY-01

改一条共享规则，怎样证明网页和训练都没坏

共享 Runtime 把 Browser、Headless 和训练接到了同一套规则上，但“同一套规则”不能靠一个大测试命令证明。语法、局部行为、标准 Action、跨宿主一致、恢复、真实浏览器和机器玩家质量分别面对不同风险，需要分层门禁。

阶段判断：不需要再造一套独立测试框架；先冻结“改动类型 → 必须通过的证据层”。共享 Runtime 改动的硬门禁应是：同一 checkpoint、同一 seed、同一决策序列，在 Browser 与 Headless / Training 得到相同 committed state、合法选择、事件与 RNG journal。Chrome 终局只证明浏览器集成，AI 分数只证明策略质量，两者不能互相替代。

七层证据，各自只回答一个问题

G0

静态与归属

node --check、依赖图、旧入口零调用、模块归属检查。

不能证明规则行为正确。
G1

领域行为

纯规则函数、Effect executor、边界值、失败分支与不变量测试。

不能证明装配后可执行。
G2

Runtime 契约

从同一 checkpoint 枚举全部 legal actions 并逐个 fork 执行；验证原子提交与 fail-closed。

不能证明两个宿主一致。
G3

跨宿主一致

固定 seed 与决策 trace，对比状态、choices、events、RNG journal。

不能证明中断后可续跑。
G4

恢复与重放

稳定点及 awaiting-decision 存档恢复；同 trace 续跑并对比结果。

不能证明真实 DOM 可操作。
G5

真实 Chrome

脚本装配、事件绑定、Decision UI、投影刷新与完整终局 smoke。

不能证明 AI 会玩得好。
G6

策略与性能

固定评估协议、模型版本、多 seed 分布、吞吐与 worker 恢复。

不能反向替代规则正确性。
由左到右不是重复覆盖，而是逐层补齐前一层看不到的风险。

什么改动必须过哪些门

改动类型最低门禁关键证据不要求每次都做什么
纯规则 / 新标准 EffectG0 + G1 + G2正反例、边界值、合法 Action 可执行、失败不污染 committed state。若不改宿主、状态格式或 RNG，不必每次跑策略评估。
Action / Effect / State RuntimeG0–G4G3 跨宿主 parity 为硬门禁；涉及浏览器装配时再加 G5。“Node 全绿”不能免除 parity 与恢复证据。
Browser View / Decision UIG0 + 代表性 G2 + G5真实 Chrome 操作代表性选择、恢复 pending UI 并完成一局。UI smoke 不承担 AI 分数验收。
Headless / Training AdapterG2 + G3 + G4 + G6规则 trace 一致、checkpoint 连续、worker crash / timeout 恢复、吞吐基线。吞吐提高不能掩盖规则输出漂移。
Policy / 模型 / 训练参数接口 G2 + 独立 G6合法选择与执行接口先正确，再用冻结协议比较多 seed 分数分布。“能终局”不等于“策略变好”。

当前基础够用，缺的是统一门禁映射

已有证据资产

  • 当前工作树可找到 137 个 JavaScript 测试文件与 1 个 Python 训练测试文件；领域、app、headless、state、AI 和 training 已有较广覆盖。
  • 已有固定 seed、RNG journal、checkpoint / replay、recovery、legal-action 与 worker resilience 测试。
  • tools/run_ai_autobattle_browser.js 已能通过 CDP 驱动真实 Chrome。
  • 现有 proof obligations 已提出全合法动作 fork、旧入口零调用、Chrome smoke 与 worker A/A、A/B/A。

仍然缺少的机械约束

  • 测试很多但按历史 issue 分散，改动边界尚不能自动映射到必跑 Gate。
  • 跨宿主 parity trace、恢复矩阵与真实 Chrome 还没有一个稳定的分层入口。
  • Chrome “能完成一局”、启发式/RL “分数达标”过去容易混成同一个 AI 验收。
  • 全量 Node 适合回归兜底，但不能单独证明旧路径清零、Browser 装配或训练确定性。

本轮只需确认三点

01

按改动类型选 Gate，不要求所有改动每次跑全部证据

每个实现 issue 在开工时声明触及的边界和必跑 Gate，收口时逐项给证据；全量回归仍保留，但不替代专项证明。

02

共享 Runtime 改动必须通过固定 seed 跨宿主 parity

比较 committed state、合法选择、事件和 RNG journal；只比较“最终都能结束”不足以发现中途语义漂移。

03

浏览器集成与机器玩家质量分开验收

G5 回答页面是否真正可玩，G6 回答策略是否达到基线；训练吞吐与 worker 恢复也单列，不再用单局终局代替。

证据范围：randomizer/**/*.test.js、randomizer/training/pytorch_trainer.test.py、docs/implementation-proof-obligations.md、docs/standard-action-contract.md、docs/rl-headless-env.md、tools/run_ai_autobattle_browser.js。本轮只冻结质量门禁边界，不新建测试框架，也不启动代码迁移。
