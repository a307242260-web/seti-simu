SETI / ARCHITECTURE NOTE / EXTENSION-01

下一个边界：卡牌、科技、公司、外星人怎样接入共享规则

前几轮已经确定了 Standard Action、Effect Session、StateStore、Browser / Training Host 和机器玩家边界。现在要回答的是：新增一种规则内容时，允许它接入哪些标准端口，如何避免再增加 app 专用流程、UI 专用状态和 AI 专用 resolver。

阶段判断：不应把卡牌、科技、公司和外星人强行改成同一种数据格式；它们的规则形态不同。真正应该统一的是接入端口：定义、Action、事件 Hook、Effect / Decision、State。所有规则执行仍进入同一个 Effect Session；DOM 展示和 Policy 估值是外延，不进入规则模块。

规则模块允许做什么、不允许做什么

允许的纯规则能力

  • 读取只读 snapshot / session context，枚举或校验标准 Action。
  • 响应标准事件，例如 setup、after_launch、after_play_card、round_start。
  • 返回 Effects、DecisionEffect、事件和日志；通过 working state 修改自己的状态 slice。
  • 声明稳定排序、冲突规则、schema/invariant 和存档 migration。

禁止的新旁路

  • 直接读取或修改 DOM、overlay、按钮、localStorage、AI controller。
  • 自己维护 pending / continuation，或回调 app runtime 继续流程。
  • 绕过 Effect Session 直接写 committed state、换玩家或提交 history。
  • 为了新内容复制一套 Browser executor、Headless resolver 或 Policy 合法性。

现有总控已经覆盖什么

SETI-56 Action

统一动作注册、合法候选和执行入口；扩展模块只能贡献 Standard Action，不自建宿主动作。

SETI-62 / 70 Flow

统一 Effect executor、Decision、事件 hook 顺序与长尾公司/外星人流程；这是扩展执行的唯一 runtime。

SETI-71 State

统一权威状态 slice、版本、schema、migration 与 compare-and-commit；扩展状态不能成为第二真相。

SETI-72 / 78 Browser

迁移公司和外星人 Decision 呈现；规则模块不携带 DOM renderer。

SETI-90 Policy

机器玩家共用标准规则输入；启发式估值与 Learned encoding 是可选外延。

尚未冻结 Extension Contract

缺少统一 manifest / registry、hook 事件表、注册冲突与版本兼容规则，以及“新增内容不准碰 app”的机械验收。

建议的三个贯穿 proof case

案例要证明什么为什么选它
一张组合式新卡只增加 definition 并复用现有 Effects，即可同时在 Browser / Headless 生效。验证最小数据驱动路径。
一个跨域公司被动通过标准领域事件产生 Effects，不修改发射/科技/打牌实现，也不新增 app hook。验证当前最分散的 hook 问题。
一个含多次选择的外星人能力物种模块只注册 state、hooks、effects、decisions；Browser renderer 可替换，Policy 走同一 choices。验证最复杂专用 runtime 能否消失。

本轮只需确认三点

01

统一端口，不统一四类内容模型

卡牌可以继续偏数据/DSL，外星人可以继续是独立规则模块；它们只需遵守相同的 Action / Hook / Effect / State 接口。

02

被动与触发统一走事件 Hook → Effects

新增公司、科技或外星人不得在已有领域函数里继续插专用分支；hook 只产生 Effect，由 SETI-62 决定顺序并原子提交。

03

规则、展示、估值三者拆开

共享规则模块不带 DOM 或 Policy；Browser Rule View 与 Policy Extension 只通过稳定 definition/effect/decision id 配对。

证据范围：randomizer/game/cards/**、game/tech/**、game/industry/**、game/aliens/**，对应 app runtime，以及 SETI-56 / 62 / 70 / 71 / 72 / 78 / 90。此报告只冻结扩展接入边界，不设计具体插件框架，也不启动规则迁移。
