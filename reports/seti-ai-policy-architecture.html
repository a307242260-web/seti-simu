机器玩家：启发式版本与强化学习版本

机器玩家分为两类实现。启发式版本是可独立使用的稳定基础玩家；强化学习版本同时包含外延训练链和玩家版推理运行时。两者不复制规则，统一复用已经标准化的 Action、Decision、Flow 与 State。

目标结构

启发式机器玩家正式基础版本
  • 独立封装估值、目标、规划与选择规则
  • 保持当前已有的一定游戏水平
  • 可直接作为玩家对局中的机器席位
  • 也可作为训练示范与固定评测对手
强化学习 AI训练 + 正式游玩

机器席位决策
公共机器玩家接入接收当前决策上下文与合法选项；返回一个标准 Action / Decision 选择
共享游戏内核Standard Action · Decision · Effect Flow · StateStore · Rules · Replay
规则边界：两类机器玩家只负责选择；合法性、执行、pending / effect 流转、状态提交和确定性结算全部由共享标准组件负责。

职责边界

能力负责内容明确不负责
启发式机器玩家候选特征、价值估计、目标、浅规划、选择偏好、诊断不枚举第二套合法动作，不执行规则，不直接写状态
RL 训练侧采样、轨迹、BC / PPO、optimizer、评测、checkpoint 与模型发布不进入玩家版进程，不成为浏览器规则依赖
RL 玩家推理侧加载版本化模型、编码 observation、应用 mask、输出一次选择不训练模型，不静默调用启发式代选单步动作
公共机器玩家接入席位类型、决策点调用、版本校验、提交选择、运行诊断不包含具体策略，不定义规则合法性
共享游戏内核唯一 Action / Decision、Flow、State、Rules 与 replay 事实不依赖启发式评分、模型结构、worker 或 optimizer

  • 后续重点是给它稳定的机器玩家接口，不改现有策略水平
  • app/ai/** 中执行、pending 与调度需要按新边界拆开

  • 训练产物还需要稳定的版本、加载与兼容契约
  • 玩家版只接 inference runtime，通过同一机器玩家接口参赛

下一轮只确认三件事

01

公共机器玩家接口

输入应包含哪些 observation、legal choices、actor / owner 与版本信息；输出如何稳定映射到 Standard Action / Decision。

02

模型产物边界

训练侧如何导出模型、schema 和版本；玩家推理侧如何加载、校验与回滚。

03

失败与切换

建议不允许单步静默 fallback；模型不可用时只在开局或加载阶段明确把整个席位切换为启发式机器玩家。

依据当前仓库实现、AI domain migration、headless env 契约及 SETI-45 已确认讨论结论整理。此页只描述 AI / 机器玩家边界，不展开训练算法与规则迁移细节。
