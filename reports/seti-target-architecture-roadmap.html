技术债优先级

优先级先解决什么为什么明确不做什么
P0完成 Effect Session 与 StateStore 的在途迁移。它们是 Browser、Headless、AI 和恢复共同依赖的主干；不先收口，其他层仍会各自维护 pending、continuation 和状态副本。不新建另一套流程引擎或状态容器。
P1让 Browser Host 和两类机器玩家只通过标准端口输入输出。当前最大风险是 UI、AI resolver 与共享规则同时拥有合法项和流程推进权。不让 AI 点击 DOM，不让 UI 直接结算规则。
P2删除旧 pending、continuation、旁路 executor 和双写 adapter。新接口“能用”不等于迁移完成；旧路径继续存在会让后续改动再次分叉。不无限保留兼容层，不以改名代替删除。
P3在所有权稳定后再拆超大文件、整理脚本加载和局部模块。app.js 与外星人 runtime 仍大，但文件大小不是当前最危险的问题。不为降行数平移巨石，不以更换前端框架作为前提。

分阶段路线

阶段 A｜现在：收完共享主干

SETI-62 + SETI-71 按现有子 issue 串行推进；完成剩余 Effect 流程、训练宿主、StateStore 切片与投影契约。Standard Action 不重做。

阶段 B｜接宿主：浏览器与机器玩家归位

SETI-72 + SETI-90 消费同一 Action / Decision / State；页面只做 Projection + Input，启发式与学习型 Policy 只做选择。

阶段 C｜收双轨：删旧入口和兼容层

逐域核对卡牌、科技、扫描、公司、外星人、回合与恢复；新旧路径不能长期并存，adapter 必须有明确删除条件。

阶段 D｜进入产品迭代

新卡牌/公司/科技/外星人优先组合已有 Effects；全新能力实现标准 Effect。AI 水平、RL 训练和页面体验分别迭代，不再改规则内核边界。

保留与停止

继续保留

  • 传统脚本加载和现有页面外观，可渐进迁移。
  • randomizer/game/** 的领域模块与已有 Effects。
  • 启发式机器玩家作为浏览器对手、teacher 和固定评测基线。
  • 以单元测试为主；改到浏览器或训练链路时只补最小专项检查。

停止继续扩张

  • 在 app.js 或领域 runtime 中新增专用 pending / continuation。
  • Browser、Headless、AI 各维护一套动作合法性或执行流程。
  • 为了理想分层再造新框架、extension registry 或第二套 runtime。
  • 把训练器、worker、优化器带进浏览器玩家端。

本轮只需确认三点

01

不再开新的大架构总控

以 SETI-62、71、72、90 为现有路线，缺口优先补到这些总控下，不重复建设。

02

推进顺序是“共享主干 → 宿主接入 → 删除旧路”

允许 Policy 独立化并行，但 Browser 正式迁移要等待 StateStore / Effect Session 稳定端口。

03

以单一所有权收口，不追求一次性完美

规则入口、流程推进、权威状态各只有一个所有者即可进入产品迭代；大文件、模块形式和视觉重构留作后续局部优化。
