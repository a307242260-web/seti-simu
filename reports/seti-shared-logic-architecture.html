4. 一个逻辑组件的共同执行流程

以「登陆」为例，两种产品只在第 1 步和最后一步不同，中间的合法性判断、状态更新与结算必须完全共用。

网页点击
或 Policy 选择输入外延
→
校验「登陆」费用、探测器、目标
→
执行「登陆」唯一状态更新入口
→
自动结算奖励、触发、任务
→
产生新可见状态屏蔽私有信息
→
网页渲染
或返回 Observation输出外延

5. 当前代码实际落点

现在已经有完整动作语义，但尚未全部统一成同一种组件形态。

当前区域承载内容事实判断
randomizer/game/actions/**发射、环绕、登陆、研究科技4 个顶层动作已经有统一的 canExecute / execute 组件接口。
randomizer/game/abilities/**发射、移动、环绕、登陆、放数据、分析、科技步骤、扫描步骤原子能力已较集中，但还不是全部顶层业务动作的统一入口。
randomizer/app/** 与 app.js扫描、分析、打牌、PASS、移动、交易、公司能力、弃牌角标、数据放置、符文族、回合结束功能可用，但业务规则与流程/交互编排仍有不同程度混合。
randomizer/app/headless-contract.js15 个顶层动作族 + 7 个条件选择族训练接口已经给出了完整语义目录，是当前组件盘点的权威清单。

当前结论

业务上：SETI 当前有 15 个顶层逻辑组件，其中玩家最核心的就是发射、环绕、登陆、扫描、分析数据、研究科技、打牌和 PASS。

代码上：只有发射、环绕、登陆、研究科技已经统一进入 game/actions；其余动作虽然已实现并进入训练动作契约，但组件边界仍散落在不同 app runtime 中。

证据基线：当前工作树中的 headless-contract、game/actions、game/abilities 与 app runtime。报告不描述未来重构方案。
