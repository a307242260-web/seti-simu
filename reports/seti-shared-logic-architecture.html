共享逻辑组件
现在是什么

SETI 分成给人玩的版本与训练机器玩家的版本。两者的核心差异在逻辑组件外延；进入逻辑组件后，应复用同一套状态流转与规则结算。

本轮范围 只画「当前共享逻辑组件」：输入如何进入、状态如何流转、规则在哪里执行、输出如何投影。UI 设计、policy 算法、训练基础设施暂不展开。

现在已经共享什么，还没共享什么

最重要的不是目录整齐，而是两个版本是否在同一动作下得到同一状态结果。

当前架构判断

两个版本已经共享规则执行与状态流转主体，但共享逻辑还不是一个可独立实例化、只有窄接口的完整组件。

边界完成度

  • ✓共享状态变更主体
    browser 与 headless 都驱动同一 app/game 执行链。
  • ✓共享领域规则
    卡牌、科技、行动、外星人等没有训练专用副本。
  • △统一动作入口
    两端在深层汇合，但表层 action contract 仍不同。
  • △统一可见状态
    headless 有 observation sanitizer，网页仍直接读运行态。
  • △独立实例化
    headless 仍加载浏览器脚本清单并启动 app.js。
边界给人玩的版本训练机器玩家的版本是否共用
动作来源 DOM 事件、按钮、picker policy 从 legal actions 选择 外延不同
动作执行 app flow → game rule headless step → app flow → game rule 主体共用
权威状态 app runtime + domain state 同一份 app runtime + domain state 共用
结果输出 refresh / render / DOM observation / reward / replay 外延不同
可见性投影 渲染层从运行态取数 publicState / selfState sanitizer 尚未统一
