训练版本外延：一条 Env 边界，三层训练宿主

玩家版与训练版继续共享同一套游戏状态、Action、Effect/Decision 流程和规则。训练版真正新增的，不是第二套游戏，而是环境适配、采样基础设施和学习系统三层外延。

阶段判断

当前训练链已经能跑：headless env、合法动作、observation、worker/IPC、replay/checkpoint 和 PyTorch 训练器都已有实现或在途；但这些能力仍建立在迁移中的 app runtime 上，尚未收敛到刚刚确定的 Standard Action / Effect Session / StateStore 稳定端口。

03

一个训练 timestep 应怎样闭环

Policy 只在真正存在多个非等价选择时介入；UI 确认、唯一选项和内部效果推进都不应占用模型决策步。

共享内核停在决策点顶层 Action 或 DecisionEffect；owner 已确定。
→
生成 Observation公开信息 + 当前玩家私有信息；不泄漏未来与对手隐藏信息。
→
枚举 Legal Actions稳定 actionId / maskIndex；不带 heuristic score。
→
Policy 选择输出一条标准 Action / Decision；记录 logits、log-prob、value。
→
执行并自动 drain规则内核推进确定性 Effect，直到下一决策点或终局。
→
返回 StepResult新 observation、legal actions、reward、done、replay event。
下一决策点继续循环；终局时 flush 最后结算与 replay 后才 done = true

06

当前三个推进风险

问题不是“训练有没有代码”，而是训练代码和共享内核迁移怎样避免互相拖垮。

协议稳定、底层迁移

headless-env.js 已被 trainer / worker 消费，但底层正从 app runtime 迁向 Action / Session / StateStore。应冻结 Env v1 外部行为，用 adapter 吸收底层变化，避免训练器跟着重写。

两套“条件选择”再次分叉

SETI-39 当前失败点与 SETI-60 刚完成的 decision family 重叠。若 trainer 局部补 heuristic fallback，会重新制造训练专属规则；修复必须落在共享 Action / Decision 合法性。

训练指标掩盖架构缺口

吞吐、100 局终局和 PPO loss 只能证明链路局部工作，不能证明 hidden information、replay parity、working-state observation 与最终共享边界正确。

07

这轮建议只拍板三个边界

先确定训练宿主的外沿与共享内核接口，不在本轮讨论网络结构、超参数或最终部署。

1

“训练版本外延”是否包含三层？

建议包含：Env Adapter、Sampling Infrastructure、Learning System。三层都属于训练产品，但只有 Env Adapter 直接接共享游戏内核。

2

Reward 的事实与优化怎样分工？

建议共享内核只产出规则事件、资源/分数变化和终局结果；Env 形成稳定 reward schema；Trainer 可配置 shaping，但不得把 shaping 写回规则与 observation。

3

现有训练与内核重构怎样并行？

建议 SETI-39 继续完成最小 BC/PPO 实验；同时把修复限定在共享 decision 合法性，不扩大 Env v1；新架构总控只负责协议冻结和迁移，不重做训练算法。

08

推进建议：先冻结端口，再建立训练宿主总控

保留现有训练实验的执行连续性，同时给后续架构迁移一个清楚的 owner。

建议顺序

  1. 本轮先确认上面三个边界。不立即改 reward、网络或 worker。
  2. SETI-39 继续处理当前最小阻塞。research_tech_select 修复必须复用 SETI-60 的标准 decision 语义，禁止 trainer / heuristic fallback。
  3. 确认后新建“训练宿主外延总控”。冻结 Env v1、Observation、Action encoding、Reward/Done、checkpoint/replay、worker/trainer 分层与迁移 proof；关联 SETI-14/39，但不搬动正在执行的任务。
  4. 选研究科技作为第一条贯穿验收链。同一 Action / Decision trace 同时证明浏览器、headless、worker 与 trainer 数据一致。

明确暂不做

不另起第二套 simulator；不让 heuristic 进入 PPO rollout；不把 UI confirm 当 policy action；不在共享内核未稳定时扩展 Env v2；不以 PPO 能更新或单局高分替代协议 parity。
