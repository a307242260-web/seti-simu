SETI 新架构：一个规则内核，两个宿主

新架构把浏览器原型从“UI 流程即规则”的大型闭包，收敛为由权威状态、语义 Action、 Effect Session 和宿主投影组成的可验证内核。浏览器与训练环境不再各自解释游戏。

基于当前代码与设计文档 无构建步骤 Browser / Simulation 同源 Fail-closed

Executive judgment

设计的核心不是“拆文件”，而是“冻结所有权”

当前代码已经形成清晰的单向依赖：宿主只能提交输入和读取投影；规则层独占合法性与变更； StateStore 独占已提交事实。这样才能让浏览器玩家、机器席位和训练 Policy 在相同的规则边界上行动。

一句话概括： 玩家或 Policy 只选择“做什么”；Standard Action 判断“能不能做”；Effect Session 负责“如何完整结算”； StateStore 决定“能否成为新事实”；Browser / Simulation 只决定“如何看见与如何输入”。

Design principles

五条架构原则

1

单一权威状态

规则事实只存在于 committed snapshot，或基于其版本创建的唯一 Session working copy。

2

语义输入

输入使用稳定 Action / Decision descriptor，不携带 DOM、闭包、估值或按钮文案。

3

事务化结算

一个动作的 effect、触发、选择、日志、撤销与不可逆屏障在同一 Session 内推进。

4

宿主无规则

Browser 负责投影与交互，Simulation 负责 observation 与 step；两者都不拥有第二套规则。

5

默认拒绝

旧 schema、stale 版本、越权 owner、未知 family 和非法 Policy 输出都 fail-closed。

6

同源可验证

浏览器与训练共享 state、Action、Effect、Policy 端口，parity 可以由相同 checkpoint 证明。

Action lifecycle

一次动作如何成为新事实

所有入口最终经过同一条事务链。任何一步失败都不会把半成品写回 committed state。

01读取投影BrowserProjection 或 observation 只暴露当前 viewer 可见信息。
02选择 Action人类输入或 Policy 从 legal descriptor 中选择语义 actionId。
03重新校验检查 schema、actor、stateVersion、decisionVersion、候选成员关系。
04打开 Session从 committed snapshot 创建带 baseVersion 的 working copy。
05推进 Effect按 direct → trigger → deferred 排序结算；多选时停在 DecisionEffect。
06CAS 提交完整通过不变量后，compareAndCommit 原子更新 stateVersion。
07刷新派生物生成新投影、事件、history、log、replay；机器席位获得新代次。

确定性 drain

零选项按规则失败或落空；唯一等价项自动推进；两个以上非等价项必须形成新的策略选择点。drain 有步数上限。

选择是显式边界

DecisionEffect 带 owner、decisionVersion 和 choices。浏览器按钮与 AI 都只能提交 choice，不能用回调旁路执行。

State model

状态生命周期：事实、草稿与派生物

Committed snapshot

唯一权威事实；深拷贝、冻结、带 stateVersion。

→
Session working copy

基于 baseVersion 的事务草稿；持有 effect queue 与 journal。

→
新 committed snapshot

只有 CAS 与全部不变量通过后才发布。

规则状态

match / turn / players / solarSystem / pieces / planets / data / cards / tech / aliens / finalScoring

事务记录

actions、decisions、effects、events、RNG、history、logs 与 replay 随 Session 前进。

非权威派生物

Browser ViewState、行动日志视图、observation、报告与训练 checkpoint 不得反向覆盖规则事实。

Host separation

同一个内核，两种运行形态

Browser Host

  • 把 committed/session 状态投影成按 viewer 隔离、深冻结的 BrowserProjection。
  • 把点击、选择、拖拽转换为 Action / Decision submission。
  • 管理 ViewState、DOM renderer、Browser Services 与本地恢复交互。
  • 人类席位与机器席位共用 BrowserInputAdapter 提交边界。

Simulation Host

  • 直接装配 simulation-rule-composition 与公共 rule-composition。
  • 向训练端提供 reset / observe / legalActions / step / checkpoint / replay。
  • 把相同 descriptor 投影成 RL envelope，不创建第二套 candidate identity。
  • 不加载 index.html、app.js、DOM 或 Browser globals。
Parity 的含义：不是两个实现“看起来行为相同”，而是从同一 checkpoint 执行同一 actionId，得到一致的 committed state、事件、RNG cursor 与下一决策边界。

Machine player

AI 是决策者，不是自动化旁路

新架构把 AI 从浏览器 resolver / candidate / executor 链条中抽离。Policy 只能消费公共 DecisionContext， 并返回 legal set 中的 actionId；Machine Player Host 负责异步请求安全，规则内核负责最终重验与执行。

DecisionContext

observation + legal descriptors + owner + versions

→
PolicyDecision

actionId + provenance；不携带 executor 或 DOM callback

→
公共输入端口

提交前重验；未知、stale、超时、重复均 fail-closed

Policy Port

定义版本化请求、响应、validator 与请求失效语义。

Heuristic Policy

只读估值和稳定排序，可供浏览器席位、teacher 与冻结 opponent 共用。

Machine Player Host

固定席位、代次、deadline、取消、去重；不执行游戏规则。

Assessment

当前完成度与后续关注点

已经建立的生产边界

22 个 Standard Action family、统一 Effect Session、StateStore CAS、BrowserProjection、Simulation Env、公共 Policy Port 与 Machine Player Host 已在代码中形成对应 owner。

旧旁路正在收口

浏览器机器席位不再暴露 legacy candidate / selector / pending automation；恢复与输入只接受当前 schema 和公共端口。

Composition root 仍然偏大

randomizer/app.js 当前仍约 4,610 行。后续拆分应继续以 owner 和显式 context 为依据，不能只把函数搬到新文件。

真实装配证据不可省略

跨模块迁移除了 Node 行为回归，还必须保留真实 Chrome smoke、固定 seed 完整局、checkpoint fork 与 Browser/Simulation parity。

演进准则：未来新增机制时，先回答“状态由谁拥有、Action identity 在哪里生成、Effect 在哪个 Session 结算、宿主只看到什么投影、失败是否无副作用”，再决定文件放置。
