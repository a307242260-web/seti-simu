一项行动内部怎样接棒

这是当前最关键的衔接模型：主行动不是一次函数调用结束，而是不断在「规则执行」与「等待选择」之间往返，直到 effect chain 完成。

Action / Ability

校验合法性、支付成本、改变权威状态、产出事件与后续 effects。

  • 主行动 / 快速行动 / 卡牌
  • 历史 session 在这里开始
  • 结果可同步，也可要求后续交互

actionEffectFlow

统一的顺序效果队列，保存当前 index、owner、history source 与延迟结算数据。

  • 执行当前 effect
  • 自动奖励可连续推进
  • 复杂 effect 暂停并转交 pending

pendingState.*

05

三条横切流程

它们不属于某一个回合阶段，却在每次接棒时决定系统是否能继续运行。

Crosscut A · Render

渲染与交互锁

refresh.js 已提供板面、玩家面板和行动状态三个聚合刷新入口。

但领域 flow 仍经常手工调用若干 render/update 函数，pending 打开或关闭时需要同步 CSS focus、按钮禁用与 overlay。

状态变化 → 多个 render/update → AI 调度
Crosscut B · Automation

AI 与 headless 推进

AI automation 先处理 pending，再处理 effect，再选择顶层行动；headless 也枚举顶层与条件动作。

好处是共享同一权威状态；代价是自动化层必须认识大部分 pending 类型和它们的优先级。

pending owner → legal conditional action → execute → next boundary
Crosscut C · Recovery

日志、存档与恢复

稳定边界会生成 recovery snapshot 并延迟写入 localStorage；处于 effect、pending、history session 中时拒绝覆盖稳定存档。

恢复时清空所有 transient state，再整体刷新 UI。这是安全的，但意味着当前模型只能恢复到确认边界，不能恢复任意半流程。

stable boundary → snapshot → clear transient → refresh
06

当前最影响演进的四个问题

按对整体流程衔接的影响排序；不是本轮直接重构清单。

01
pending 不是单一状态机
52 个字段中混有互斥选择、并行队列、状态锁、序号和 continuation。调用方通过「哪些字段非空」推断当前流程位置。
影响：新增规则容易漏 UI、AI、headless 或恢复清理。
02
完成协议不统一
有的子流程调用 completeCurrentActionEffect，有的调用领域 continuation，有的清字段后再显式刷新/调度。
影响：「选完但没继续」「继续两次」是典型跨模块风险。
03
app.js 仍是总接线板
模块已大量拆出，但 composition root 仍持有权威状态、owner 解析、效果完成、恢复、刷新和公共 API 装配。
影响：局部模块边界清晰，跨流程边界仍难单独测试。
04
渲染是命令式副作用
刷新 helpers 减少了重复，但流程完成后仍需记住要刷新哪些面板、按钮、overlay 和 AI step。
影响：规则正确时仍可能出现界面或自动化停滞。

阶段判断与下一讨论点

本轮先确认事实和边界，不在报告里偷跑成重构方案。

阶段判断

SETI 当前有一条真实、完整、可运行的游戏流程主链。轮次推进、效果队列、历史事务、终局结算都已经有明确骨架；因此不应该另起炉灶重写总流程。

真正需要收口的是「接棒协议」：顶层行动怎样进入 effect，effect 怎样打开 pending，pending 完成后怎样恢复，最终怎样提交、刷新、保存并交给 AI/下一玩家。

建议下一轮只讨论三个边界

  • 流程状态：当前究竟处于 setup / idle / action / effect / choice / turn-end / game-end 哪一阶段？
  • 统一完成信号：一个 pending 完成后，应该返回什么结果，由谁决定下一跳？
  • 稳定边界：哪些点允许提交 history、保存快照、切玩家与交给自动化？

本轮不建议做

不直接引入新框架，不把所有 pending 一次性改成单一大枚举，也不把领域 UI 全部强推成声明式渲染。

理由

现有规则复杂度高、覆盖面广；应先定义接棒契约，再按高频链路逐段迁移，否则会把已跑通的领域流程重新搅在一起。
