权威游戏状态：现在是一组事实，不是一个整体

当前实现已经有明确的规则事实来源，也能稳定保存和供 headless 训练复用；但它由 12 个顶层可变切片组成，部分切片混有 UI / 可重算信息，尚未形成带版本、事务边界与确定性保证的统一 GameState。

01

权威边界现在在哪里

app.js 创建并持有状态对象，各领域模块通过显式 context 取得引用并直接修改。createGameRecoverySnapshot() 把 12 个切片组合起来，是目前最接近“权威状态根”的实际定义。

12进入恢复快照的核心切片事实上的持久化根
1浏览器与 headless 共用的规则实现没有第二套模拟器状态
0统一 GameState aggregate没有统一 schema / stateVersion
Stable仅稳定边界允许本地保存执行中状态暂不落盘

运行态（不应直接成为棋局事实）

  • pending / 当前 effect flow
  • action & quick history session
  • AI 控制、日志草稿、overlay / picker
  • DOM、渲染缓存、调试模式
→

12 个 core_state 切片

  • 太阳系、玩家、轮次、火箭、星球
  • 星云数据、科技、卡牌、任务卡
  • 外星人、终局计分、开局选择
  • 由 app.js 统一装配，但非单对象
→

派生视图

  • legal actions / observation
  • 计分摘要、AI 估值
  • 渲染结构、可见位置
  • 原则上可由状态重新计算
恢复快照 = 12 个 core_state + aiControl　｜　恢复前先清空 pending、effect、history、选择 UI，再整对象回填并全量刷新
准确说法：当前不是“没有权威状态”，而是“权威状态已存在，但它的边界由快照代码和调用约定定义，尚未由独立数据模型定义”。

03

真正需要区分的三类状态

现在代码中已经隐约存在这三层，但类型和所有权没有把它们强制隔离。

A · DURABLE FACTS

已提交的棋局事实

如果关掉界面再打开，仍应原样成立；任何合法行动与计分都以它为输入。

  • 玩家资产、牌与科技
  • 棋盘位置、轮次与顺位
  • 痕迹、终局板块、数据控制
  • 未来应成为 CommittedGameState
B · SESSION FACTS

当前 Effect Session 的工作事实

执行中必须真实可见，但只有整条队列成功结束后才升级为已提交事实。

  • workingState
  • effectQueue / cursor
  • DecisionEffect 的输入等待
  • 撤销边界与不可逆标记
C · RUNTIME / PROJECTION

宿主运行态与派生视图

不应改变规则结论；可在恢复后重建，或由玩家版/训练版分别拥有。

  • overlay、picker、选择高亮
  • AI policy / 自动调度
  • legal actions、observation
  • 日志文案、DOM 和渲染缓存
04

现状的 5 个主要问题

这些问题不是说当前规则不可用，而是说明它还不足以直接作为玩家版与训练版长期复用的稳定契约。

1
没有统一 schema
12 个变量名 + 快照拼装顺序就是事实契约；没有独立 GameState 类型、schemaVersion、stateVersion 和集中校验。
新增状态容易漏存、漏恢复、漏 observation
2
跨切片原子性靠流程保证
拿科技、环绕/登陆、卡牌移动等都会同时修改多个对象；中途失败是否留下半成品，主要由调用顺序和 history 命令约束。
正由 Effect Session 重构补齐
3
核心切片不够纯
科技/卡牌直接包含 ui，火箭含展示文案，任务卡状态可由别的事实重算；持久化边界大于规则边界。
玩家 UI 与训练宿主难以真正解耦
4
确定性状态仍有闭包外溢
模块级自增序列、缓存与 Math.random 仍存在。headless 通过每次 reset 重建模块注册表与 seed RNG 来清理，而不是只恢复 core_state。
单靠 12 切片尚不能保证任意点确定性续跑
5
只能保存稳定边界
本地保存明确拒绝 pending、effect、日志草稿或 history session 活跃的时刻；恢复还会先清空全部瞬时流程。
当前存档表达的是 committed snapshot，不是 session checkpoint
05

与 Effect Session 的自然衔接

无需推翻现有领域状态。重点是给这批状态增加一个明确容器，把执行中副本和宿主视图区分开。

CommittedGameState

纯规则事实；带 schemaVersion / stateVersion / rngState；玩家版与训练版共同读取。

→

EffectSession

从 committed 克隆出 workingState；队列顺序执行，DecisionEffect 可暂停；完成后原子替换 committed。

→

Host Projections

浏览器生成 UI view，训练端生成 observation / legal actions；都不能反向成为规则真相。

建议保留的资产：现有 12 切片的领域拆分大部分是合理的；改造重点不是合并成一个巨型平面对象，而是用一个根对象明确纳管、净化字段，并建立唯一提交入口。
06

阶段结论与下一项决策

这份现状足以进入状态模型的目标边界讨论，但还不代表应立即另开一轮大重构。

阶段结论

  • 当前权威状态的事实根，是恢复快照列出的 12 个切片。
  • 它能支撑当前浏览器存档，并已作为 headless core_state 的基础。
  • 其中真正的长期规则状态约 8–9 组；科技/卡牌需剥离 UI，任务卡状态应评估改为派生索引，开局选择只在 setup session 内权威。
  • 未来统一根应管理领域切片，而不是取代领域切片。

建议下一步只拍板一件事

是否确认：以现有 12 切片为迁移起点，目标上拆成 CommittedGameState + EffectSession + HostRuntime/DerivedView 三层？

确认后，再在 SETI-62 中定义根 schema、首批净化字段和兼容迁移顺序。
