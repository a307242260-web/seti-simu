SETI / ARCHITECTURE NOTE / LIFECYCLE-01

Game Lifecycle 与流程衔接：不是两套系统

“流程衔接”是已经确认的 Effect Session 总体方案；“Lifecycle”只是其中在 setup、换玩家、跨轮和终局边界调用的一组规则。上一版把它画成并列的新 runtime，造成了重复感，这里改为同一条执行链上的职责对照。

直接回答：不应该再建一套 Lifecycle Runtime。Effect Session 回答“这次 Action 产生的效果、选择和触发如何按顺序跑完并提交”；Lifecycle Rules 回答“跑到回合边界后，下一位是谁、是否跨轮、是否终局”。Lifecycle Rules 应被 Effect Session 调用，结果写入同一个 working state，并随同一次 session 原子提交。

用一次 end_turn 看两者区别

1. Standard Action

  • 玩家或 Policy 提交 end_turn
  • 校验 actor、版本和当前合法性
  • 只负责表达“我要结束回合”
SETI-56 / SETI-59
→

2. Effect Session（流程衔接）

  • 打开 working state 与 journal
  • 执行回合末揭示、收入、清理等 hooks
  • 遇到选择就暂停并恢复
  • 全部完成后才允许 commit
SETI-62 / SETI-70
→

3. Lifecycle Rules（其中一段规则）

  • 根据 PASS 集合选择下一 actor
  • 判断继续本轮、进入下一轮或终局
  • 返回结构化 transition / events
  • 仍由 Effect Session 执行后续 hook 并提交
当前仍在 app/turn-flow.js

尚未明确覆盖的 8 个缺口

缺口当前代码 / 契约现状应归入哪里
1. 建局工厂startNewGame() 同时做 reset、随机化、发牌、AI 配置、UI 刷新和存档；SETI-62 从 Action session 开始，没有定义 config + seed → 初始 state。独立 GameFactory 边界；与 StateStore 初始化配合，不属于 Effect Queue。
2. Setup 完成条件setupSelectionState 独立存在；何时从初始选择进入首个正常 actor 没有进入 SETI-62 标准阶段。新增 setup session / transition proof case，仍复用 DecisionEffect。
3. 宏观 phaseturn state 有 round / actor / passed / gameEnded，但没有统一 setup / playing / finishing / ended phase；调用方仍通过多个字段组合推断。SETI-71 的 turn/match schema 与 invariant。
4. 下一 actor / 跨轮 / 终局纯规则advanceTurnAfterPlayerAction()、beginNextRound()、finishGameAfterFinalPass() 仍在 app/turn-flow；SETI-70 说迁 PASS/回合末，但没冻结谁是这些规则的唯一 owner。作为 SETI-62/70 的 lifecycle transition executor；纯函数落 game core。
5. 跨阶段 hook 顺序PASS 收入、外星揭示、round-start 公司收益、原教旨收入目前手工串接；只有 continuation，没有统一 hook 顺序契约。SETI-62 Effect Queue / hook registry；Lifecycle Rules 只返回要进入哪条边界。
6. 原子换人当前先提交 history、清 pending，再调用 turn advance，随后执行 round-start hook 和刷新；不能证明所有步骤与 actor 切换属于同一次 StateStore commit。SETI-62 + SETI-71 的 end_turn 原子 proof obligation。
7. 恢复与 terminal 闭合恢复多在稳定边界；终局还要等终局标记、奖励、弃牌和 replay flush。何时 legalActions 必须为空没有统一 core 判定。SETI-62 session inspect + SETI-71 committed phase + RL/browser parity。
8. Host 启动去浏览器化headless reset 仍加载 browser bundle 并调用 startNewGame()；Browser / Training 尚未共享独立 runtime 创建入口。SETI-72 / training host 接入；消费同一 GameFactory 与 session runtime。

怎么处理这些缺口

不建新的 Lifecycle 总控

宏观 transition 是流程衔接的一部分，另建并列 runtime 会与 SETI-62 争抢 session、turn-end 和 commit 所有权。

补 SETI-62 / SETI-70

明确 lifecycle transition executor、hook 顺序、setup / end_turn proof cases 与原子提交验收。

补 SETI-71

统一 committed macro phase、actor / round / terminal invariant，并让 lifecycle transition 成为唯一写入口。

补 SETI-72 / Training Host

去掉 Host 自己的轮转与 terminal 判断；最终都从相同 runtime boundary 请求下一决策。

GameFactory 单独评估

建局不是 Action 后 Effect Session；它可能需要独立子 issue，但应先确认 config、seed、初始 state 和 setup session 边界。

SETI-5 保留为历史基线

它只完成 app.js 的无行为拆分，不把“文件已拆”误判为共享 game core 已完成。

修正后的结论

01

没有第二套流程系统

Lifecycle 只保留为纯规则/transition executor，嵌入 SETI-62；Effect Session 仍是唯一流程衔接 runtime。

02

现有总控补边界，不重复建单

SETI-62 补执行与原子性，SETI-71 补 committed phase / invariant，SETI-72 与 Training Host 补宿主接入。

03

唯一可能独立的是 GameFactory

config + seed → 初始 state 与 setup session 不属于普通 Action 后流程；先把契约讲清，再决定作为 SETI-62 子项还是单独实施 issue。

修订依据：SETI-62 完整范围、既有流程衔接报告、`turn-flow.js`、`turn-end-flow.js`、`headless-env.js`、StateStore turn slice 与 Standard Action 契约。核心修正是把 Lifecycle 从“并列 runtime”降为 Effect Session 内调用的宏观 transition 规则。
