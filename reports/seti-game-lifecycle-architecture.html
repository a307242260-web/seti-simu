Game Lifecycle：从建局到终局的共享状态机

下一块需要统一的不是某个行动，而是“一局游戏何时处于 setup、谁可以行动、何时结束当前回合、何时跨轮、何时进入终局”。浏览器与训练版应共享同一生命周期事实；UI、自动调度和训练循环只消费生命周期给出的稳定决策边界。

当前判断：回合状态已经进入 StateStore，PASS / end_turn 已进入 Standard Action，但生命周期推进仍主要是 app 壳层。`turn-flow.js` 同时负责建局重置、随机化、发牌接线、轮次推进和终局摘要；`turn-end-flow.js` 同时负责规则副作用、history、pending continuation、渲染和下一玩家调度。headless reset 仍加载浏览器 bundle 并调用 `startNewGame()`，说明两种运行形态尚未共享一个独立 Lifecycle Runtime。

职责边界

组件负责明确不负责
GameFactory按 config、seed、规则版本创建完整初始 committed state；建立玩家、牌库、盘面和 setup 决策不读取 DOM，不调浏览器刷新，不选择玩家决策
Game Lifecycle唯一 lifecycle phase、actor、轮次、PASS 集合、跨轮与终局条件；输出结构化 transition/events不执行具体行动效果，不渲染，不调度 AI / trainer
Effect Session在 lifecycle transition 内按顺序执行 PASS、turn-end、round-end、round-start、game-end hooks；处理 Decision 与中断恢复不自行决定下一玩家、轮次或游戏是否结束
StateStore保存 lifecycle committed fact 与 session working state；compare-and-commit、恢复、版本和 invariant不定义生命周期业务规则，不持有 UI 派生状态
Browser / Training Host启动 runtime、展示或采样、在 TURN READY 请求 human/Policy action、消费 events 与终局结果不复制 turn advance，不绕过 lifecycle 强行换 actor

最小公共契约

输入命令

  • createGame(config, seed)
  • submitSetupDecision(decision)
  • completeTurn({ actorId, passed })
  • resumeLifecycle(decision)：继续被 hook 阻塞的 transition

输出边界

  • phase / round / actor / stateVersion
  • LifecycleTransition：before、after、reason、events
  • 下一项 Standard Action / Decision，或明确 terminal
  • 相同 state、command、decision、RNG journal 得到相同结果

这轮只确认三件事

01

Lifecycle 是否属于共享 game core

建议确认。浏览器与训练宿主只能消费同一 lifecycle state / transition，不再各自维护 setup、轮转和 terminal 判断。

02

副作用如何接入

建议 Lifecycle 只决定阶段和顺序，PASS / turn-end / round-start / game-end 的具体规则 hook 全部交给 Effect Session；hook 未完成时不得提交下一 actor。

03

建局边界

建议 GameFactory 负责从 config + seed 生成初始 state；Lifecycle 从 SETUP 开始拥有阶段推进。浏览器存档、动画、对话框和训练 reset 都留在各自 Host。

依据 `randomizer/app/turn-flow.js`、`turn-end-flow.js`、`headless-env.js`、StateStore turn slice、Standard Action 与现有 SETI issue 状态整理。此页只讨论 Game Lifecycle 一级边界，不在本轮设计具体 hook registry、文件拆分或迁移排期。
